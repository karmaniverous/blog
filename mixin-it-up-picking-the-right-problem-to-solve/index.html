    What is Good Code?

    Well, clearly, at a minimum good code fulfills requirements: it does what it was meant to do.

    But every grown-up developer knows there is more to it than that. Really good code is more than just a dropped plate of spaghetti that happens to stick to the wall. It also meets a coherent set of engineering principles, like:

    • SOLID - Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion
    • DRY - Don‚Äôt Repeat Yourself
    • KISS - Keep It Simple, Stupid
    • YAGNI - You Ain‚Äôt Gonna Need It (so don‚Äôt write it)
    • Law of Demeter - Only talk to your immediate friends
    • Composition over Inheritance - Prefer composition to inheritance

    Good code encapsulates the understanding that the work is not done: my good code will eventually have to talk to your good code, and it behooves us both to make that conversation as easy as possible.

    A Brief Detour into Mixins

    If you recall, we started this conversation because I ran into some documentation-related problem with my Batchable mixin implementation. But wait‚Ä¶ what exactly is a mixin?

    Before we dig into the actual problem, let‚Äôs take a moment to understand the distinction between a mixin and a class.

    A Typescript class is a way to encapsulate a complex set of behaviors and data into a single, reusable unit. The Typescript version adheres pretty faithfully to the Object-Oriented Programming (OOP) paradigm, with the caveat that ‚Äúclass‚Äù inheritance in Typescript is really prototype-based.

    Compared to a more ‚Äúpure‚Äù OOP language like Ruby or (shaddap haters) C++, a key constraint imposed by Typescript‚Äôs prototype-inheritance model is single inheritance.     For example, let‚Äôs say I have a Loggable class that knows how to log to an external logging system, and a Batchable class that knows how to batch & throttle data operations against an API.     In Typescript, I can‚Äôt do that. I can only inherit from a single class, so I have to find some way to‚Ä¶

    • Preserve the useful isolation of distinct packages of functionality into the Loggable and Batchable classes.

    • Combine them into a single class that EntityManager can inherit.

    This is where mixins come in.

    A mixin is a function that takes a class as an argument and returns a new class that combines the functionality of the original class with the functionality of the mixin. In the degenerate case, a mixin takes an empty class as its argument and returns a new class that exhibits only the mixin functionality.

    We‚Äôve already established that tools like TypeDoc can activate the capacity of self-documenting code to do really useful things like automagically generate complex API documentation.

    This prompts an observation so obvious that it‚Äôs easy to overlook: the most valuable tools come with a built-in set of constraints. In other words, the more I like what TypeDoc does with my code, more important it is to stick to code TypeDoc can actually digest.

    To make any use of a Typescript class, I need to know a great deal about its internal structure. Therefore, TypeDoc takes great pains to document the properties, methods, and types related to any class.

    The Problem Behind the Problem

    So far, I‚Äôve presented this problem from a very particular perspective: I have good reasons to do an advanced thing and am running up against the constraints of my toolbox.

    Ring a bell?

    Recall that the entire topic of mixins is only relevant because I wanted to overcome Typescript‚Äôs multiple-inheritance constraint and inherit functionality in a single class from both Loggable and Batchable.     The Composition Solution

    If extreme difficulty can be a sign that you‚Äôre on the wrong track, then what about the reverse?

    Watch this:
